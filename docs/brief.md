# Project Brief: 数当てCLIゲーム

## Introduction
このドキュメントはCLI数当てゲーム（1〜20の数を当てるミニゲーム）のProject Briefを作成するためのガイドです。以降、各セクションを順に確認し、内容を詰めていきます。必要に応じて仮置きの前提・アイデアを提示し、あなたの判断で修正します。

> 注: 本ブリーフはインタラクティブに構築中です（以降のセクションは作業中）。

## Executive Summary
本プロジェクトは、1〜20の整数を当てるシンプルなCLIゲームを短時間で学習・体験できる形で提供します。解決する課題は「初学者が小さな成功体験を得にくい」問題で、短サイクルでの実装・実行・改善を通して達成感を得られます。ターゲットはプログラミング初学者、授業/ワークショップ、CLI操作に慣れたいユーザーであり、主要な価値は即時のフィードバック（高すぎ/低すぎ/正解）による学習促進と、短時間で完成できる実装容易性です。

## Problem Statement
- 現状・ペイン: 初学者は学習題材が複雑化しがちで、短時間で「動くもの」を完成させる体験が得にくい。
- 影響: 達成感不足により学習継続率が低下。基礎概念（入出力、条件分岐、ループ、乱数）を実践で学ぶ機会が減少。
- 既存解の課題: WebやGUI題材は環境構築・依存が多く、最初の一歩でつまずきやすい。教材も分散し、即時実行が難しい。
- 緊急性/重要性: 授業・ワークショップ等で短時間に成功体験を提供するニーズが高く、導入障壁の低い教材が求められている。

## Proposed Solution
単一ファイルまたは少数モジュールで構成されるCLIアプリとして実装します。起動時に1〜20の乱数を生成し、ユーザー入力に対して「高すぎ」「低すぎ」「正解！」の即時フィードバックを返します。無効入力（非整数・範囲外）はエラーメッセージで再入力を促します。任意で最大試行回数や再プレイ確認を提供し、学習用途に合わせて難易度（範囲）を切替可能にします。実装言語は導入が容易なPython/Node.js/Goのいずれかを想定し、入出力、判定ロジック、乱数生成を分離したシンプルな構造でメンテ性を確保します。

## Target Users
- Primary User Segment: プログラミング初学者（高校〜大学初年次）
  - 目的: 基本構文と入出力を短時間で体験
  - ニーズ: 依存の少ない教材、即時実行、明確なフィードバック
- Secondary User Segment: 授業/ワークショップの指導者
  - 目的: 短時間の演習ネタ、評価しやすい課題
  - ニーズ: セットアップ簡易、拡張課題の用意（試行回数や難易度）

## Goals & Success Metrics
### Business Objectives
- 学習者の「最初の成功体験」提供（初回実行まで10分以内）
- 授業/WSでの配布と実行の容易化（環境構築5分以内）

### User Success Metrics
- 初回プレイ完了率 ≥ 90%
- 無効入力からの復帰成功率 ≥ 95%
- 平均試行回数（正解まで）: 4〜8回（適度な難易度）

### KPIs
- セットアップ所要時間: ≤ 5分
- 実装ファイル数: ≤ 3
- 依存パッケージ数: 最小（標準ライブラリ中心）

## MVP Scope
### Core Features (Must Have)
- 数当てゲーム本体（1〜20の乱数、入力、判定、フィードバック）
- 無効入力ハンドリング（非整数/範囲外）
- 正解時メッセージと終了

### Out of Scope for MVP
- 永続的ハイスコア保存
- 多言語UIの完全対応（MVPは日本語）
- 外部依存のある分析/ログ収集

### MVP Success Criteria
- 要件を満たす動作とテストケース通過
- 初学者が説明なしで遊べる簡潔なREADME同梱

## Post-MVP Vision
### Phase 2 Features
- 難易度設定（範囲/試行回数の変更）
- リプレイ（再戦）プロンプト
- 連続プレイのベストスコア表示

### Long-term Vision
- 多言語対応（ja/en）とローカライズ設計
- 学習課題の自動生成（課題カード）

### Expansion Opportunities
- Web/GUI版への展開、オンライン演習用サンドボックス

## Technical Considerations
### Platform Requirements
- ターミナルで動作（macOS/Linux/Windows）
- 標準入出力のみを使用

### Technology Preferences
- Frontend: なし（CLI）
- Backend: Python または Node.js（授業環境に合わせ選択）
- Database: なし
- Hosting/Infrastructure: なし（ローカル実行）

### Architecture Considerations
- 単純なレイヤ分割（io / logic / rnd）
- テスト容易性（比較関数を純粋関数に）
- セキュリティ: 入力検証（例外処理）

## Constraints & Assumptions
### Constraints
- 授業時間（45〜90分）内で実装・実行・解説を完了
- オフライン環境でも実行可能

### Key Assumptions
- 受講者はターミナルの基本操作が可能
- Python または Node.js が事前にインストール済み

## Risks & Open Questions
### Key Risks
- 環境差異（Windows固有の入出力挙動）
- 非同期入出力（Node.js）での例外処理漏れ

### Open Questions
- 既定言語はどれにするか（Python/Node.js）
- MVPで再プレイ確認を入れるか

### Areas Needing Further Research
- 学習効果を高めるフィードバックメッセージの最適化
- 適切な難易度レンジ（1〜20以外の設定）

## Next Steps
### Immediate Actions
1. 実装言語の決定（Python または Node.js）
2. ひな型リポジトリ構成の作成（`README.md`, 実行スクリプト, 簡易テスト）
3. コアロジック（比較関数/乱数生成/IO）の実装
4. 無効入力ハンドリングとメッセージ整備
5. 簡易テスト/デモ手順の作成
